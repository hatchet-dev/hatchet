# OpenTelemetry

Hatchet provides OpenTelemetry instrumentation as part of its SDKs. Each SDK ships a `HatchetInstrumentor` that automatically creates spans for task runs, event pushes, and workflow triggers, and propagates trace context across task boundaries.

By default, traces are sent to the Hatchet engine's built-in OTLP collector using the same connection settings as the Hatchet client.

## HatchetInstrumentor Options

Each SDK's instrumentor supports a number of configuration options. By default, the Hatchet collector is enabled and connection settings are loaded automatically from your Hatchet client config.

<Callout type="info">
  If you supply your own `TracerProvider`, the Hatchet exporter is added
  **alongside** your existing span processors — traces go to both your collector
  and the Hatchet engine.
</Callout>

<UniversalTabs items={["Python", "Typescript", "Go", "Ruby"]}>
<Tabs.Tab title="Python">

| Option                    | Usage                                                                                             |
| ------------------------- | ------------------------------------------------------------------------------------------------- |
| Custom TracerProvider     | `HatchetInstrumentor(tracer_provider=your_provider).instrument()`                                 |
| Disable Hatchet collector | `HatchetInstrumentor(enable_hatchet_otel_collector=False).instrument()`                           |
| Exclude span attributes   | `ClientConfig(otel=OpenTelemetryConfig(excluded_attributes=[OTelAttribute.ACTION_PAYLOAD]))`      |
| Include task name in span | `ClientConfig(otel=OpenTelemetryConfig(include_task_name_in_start_step_run_span_name=True))`      |
| BatchSpanProcessor tuning | `HatchetInstrumentor(schedule_delay_millis=1000, max_export_batch_size=256, max_queue_size=1024)` |

</Tabs.Tab>
<Tabs.Tab title="Typescript">

| Option                    | Usage                                                                    |
| ------------------------- | ------------------------------------------------------------------------ |
| Custom TracerProvider     | Call `setTracerProvider(yourProvider)` before `registerInstrumentations` |
| Disable Hatchet collector | `new HatchetInstrumentor({ enableHatchetCollector: false })`             |
| Exclude span attributes   | `new HatchetInstrumentor({ excludedAttributes: ['payload'] })`           |
| Include task name in span | `new HatchetInstrumentor({ includeTaskNameInSpanName: true })`           |

## Automatic Spans

The instrumentor automatically creates spans for the following operations:

| Operation             | Span Name                   | SpanKind   |
| --------------------- | --------------------------- | ---------- |
| Task execution        | `hatchet.start_step_run`    | `CONSUMER` |
| Task cancellation     | `hatchet.cancel_step_run`   | `CONSUMER` |
| Workflow trigger      | `hatchet.run_workflow`      | `PRODUCER` |
| Bulk workflow trigger | `hatchet.run_workflows`     | `PRODUCER` |
| Event push            | `hatchet.push_event`        | `PRODUCER` |
| Bulk event push       | `hatchet.bulk_push_event`   | `PRODUCER` |
| Schedule workflow     | `hatchet.schedule_workflow` | `PRODUCER` |
| Durable wait          | `hatchet.durable.wait_for`  | `INTERNAL` |

Span attributes are prefixed with `hatchet.*` (e.g. `hatchet.step_run_id`, `hatchet.worker_id`, `hatchet.workflow_name`). Trace context is automatically propagated via a W3C `traceparent` header injected into `additional_metadata`, so worker-side spans appear as children of the trigger span in your trace waterfall.
