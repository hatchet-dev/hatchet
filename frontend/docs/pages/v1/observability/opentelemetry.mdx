# OpenTelemetry

Hatchet provides OpenTelemetry instrumentation for the **Go**, **Python**, and **TypeScript** SDKs. Each SDK ships a `HatchetInstrumentor` that automatically creates spans for task runs, event pushes, and workflow triggers, and propagates trace context across task boundaries.

By default, traces are sent to the Hatchet engine's built-in OTLP collector using the same connection settings as the Hatchet client — no extra configuration required.

## HatchetInstrumentor Options

Each SDK's instrumentor supports the following configuration:

| Option                    | Python                                                              | TypeScript                                       | Go                                   |
| ------------------------- | ------------------------------------------------------------------- | ------------------------------------------------ | ------------------------------------ |
| Custom TracerProvider     | `tracer_provider=` param                                            | Set via `setTracerProvider()` before registering | `WithTracerProvider(tp)` option      |
| Disable Hatchet collector | `enable_hatchet_otel_collector=False`                               | `enableHatchetCollector: false`                  | `DisableHatchetCollector()`          |
| Exclude span attributes   | `ClientConfig(otel=OpenTelemetryConfig(excluded_attributes=[...]))` | `excludedAttributes: [...]`                      | —                                    |
| Include task name in span | `include_task_name_in_start_step_run_span_name=True`                | `includeTaskNameInSpanName: true`                | —                                    |
| BatchSpanProcessor tuning | `schedule_delay_millis`, `max_export_batch_size`, `max_queue_size`  | `bspConfig: { ... }`                             | `WithBatchSpanProcessorOptions(...)` |

<Callout type="info">
  If you supply your own `TracerProvider`, the Hatchet exporter is added
  **alongside** your existing span processors — traces go to both your collector
  and the Hatchet engine.
</Callout>

## Automatic Spans

The instrumentor automatically creates spans for the following operations:

| Operation             | Span Name                   | SpanKind   |
| --------------------- | --------------------------- | ---------- |
| Task execution        | `hatchet.start_step_run`    | `CONSUMER` |
| Task cancellation     | `hatchet.cancel_step_run`   | `CONSUMER` |
| Workflow trigger      | `hatchet.run_workflow`      | `PRODUCER` |
| Bulk workflow trigger | `hatchet.run_workflows`     | `PRODUCER` |
| Event push            | `hatchet.push_event`        | `PRODUCER` |
| Bulk event push       | `hatchet.bulk_push_event`   | `PRODUCER` |
| Schedule workflow     | `hatchet.schedule_workflow` | `PRODUCER` |
| Durable wait          | `hatchet.durable.wait_for`  | `INTERNAL` |

Span attributes are prefixed with `hatchet.*` (e.g. `hatchet.step_run_id`, `hatchet.worker_id`, `hatchet.workflow_name`). Trace context is automatically propagated via a W3C `traceparent` header injected into `additional_metadata`, so worker-side spans appear as children of the trigger span in your trace waterfall.
