# Cancellation

Hatchet lets you cancel running tasks gracefully. When a task is canceled, Hatchet sends a cancellation signal to the worker. It is your responsibility to handle this signal by cleaning up resources, aborting network requests, or otherwise stopping execution. If the task has not yet been assigned to a worker, the assignment is canceled immediately and the task never runs.

## Triggering Cancellations

Cancellations can be triggered in several ways:

- **SDK or REST API** - cancel a run by its ID using the runs client.
- **Dashboard** - cancel individual runs from the runs list toolbar or from within a specific run.
- **Concurrency strategies** - strategies like [`CANCEL_IN_PROGRESS`](./concurrency.mdx#cancel-in-progress) automatically cancel running tasks to free up slots when the concurrency limit is reached.

## Handling Cancellations

When a task is cancelled, Hatchet sends a signal to the worker, which attempts to stop the task and free its slot. Each language handles this slightly differently.

<UniversalTabs items={["Python", "Typescript", "Go", "Ruby"]}>
  <Tabs.Tab title="Python">
    For async tasks, Hatchet cancels the underlying `asyncio.Task` via the event loop. An `asyncio.CancelledError` is raised at the next `await` point, so your task is cancelled automatically. Wrap `await` calls in a `try/except asyncio.CancelledError` block if you need to run cleanup logic.

    <Snippet
      src={snippets.python.cancellation.worker.async_cancellation}
    />

    For synchronous tasks, the worker cannot interrupt the thread directly. Poll `ctx.exit_flag` inside your loop to detect cancellation cooperatively.

    <Snippet
      src={snippets.python.cancellation.worker.checking_exit_flag}
    />

    A task can also cancel itself by calling `ctx.aio_cancel()` (or `ctx.cancel()` for synchronous tasks). Any work after the cancel call will not execute.

    <Snippet
      src={snippets.python.cancellation.worker.self_cancelling_task}
    />

  </Tabs.Tab>
  <Tabs.Tab title="Typescript">
    Check `ctx.cancelled` to detect whether a cancellation signal has been received.

    <Snippet
      src={snippets.typescript.cancellations.workflow.declaring_a_task}
    />

    For tasks that make network requests or other async operations, pass `abortController.signal` to the underlying library so those operations are terminated when the task is cancelled.

    <Snippet
      src={snippets.typescript.cancellations.workflow.abort_signal}
    />

  </Tabs.Tab>
  <Tabs.Tab title="Go">
    Use the standard `ctx.Done()` channel in a `select` statement to detect cancellation. This follows the typical Go context pattern, where you can continue to propagate the context to other operations.

    <Snippet src={snippets.go.cancellations.main.cancelled_task}/>

  </Tabs.Tab>
  <Tabs.Tab title="Ruby">
    Poll `ctx.cancelled?` inside your loop to detect cancellation without forcibly killing the thread.

    <Snippet src={snippets.ruby.cancellation.worker.checking_exit_flag} />

    A task can also cancel itself by calling `ctx.cancel`. Any work after the cancel call will not execute.

    <Snippet src={snippets.ruby.cancellation.worker.self_cancelling_task} />

  </Tabs.Tab>
</UniversalTabs>

## Bulk Cancellation

You can cancel many runs at once via the SDKs, REST API, or Hatchet Dashboard. There are two approaches:

1. **By run IDs** - provide a list of task run IDs to cancel.
2. **By filters** - match runs using the same filters available in the Dashboard.
