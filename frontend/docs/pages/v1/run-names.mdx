# Naming runs with a display-name expression

By default, each run in Hatchet is labelled with a generated name of the form `<readableId>-<timestamp>` (for example, `resolve-subprocessor-info-1782146942603`). When a parent fans out many child runs in a single batch, they all share the same step id and batch timestamp, so the runs list renders several identically-named rows and you have to open each run's input to tell them apart.

You can instead give a run a human-readable **display name** by declaring a [CEL](https://github.com/google/cel-spec) expression on the workflow (and/or on individual tasks) in the workflow **definition**. Hatchet evaluates the expression against the run's input at trigger time and renders the result in the runs list, the run detail header, and the children list. Because the expression lives in the definition, it automatically names runs from **every** trigger source — manual, event, and cron — with no per-trigger configuration.

<Callout type="info" emoji="🪓">

The expression is pure CEL — the **whole string** is evaluated as one expression, consistent with concurrency and rate-limit keys. There is no `{{ }}` templating:

- **Reference the run input** with `input.<key>`, e.g. `input.customerName`.
- **Quote string literals** as CEL strings, e.g. `'Acme Corp'`.
- **Guard optional keys** with `has()`, e.g. `has(input.name) ? input.name : 'run'` — referencing a key that isn't present is an error.

</Callout>

## Workflow-level vs. per-task expressions

- A **workflow-level** expression names the run. For a multi-step DAG it lands on the DAG run; for a single-task workflow it lands on that task (the task _is_ the run).
- A **per-task** expression names an individual task of a DAG. Each step of a DAG can be named independently.
- On a **single-task** workflow the precedence is: the task's own expression, then the workflow-level expression, then the generated fallback.

<Callout type="info" emoji="🪓">

**Failures never fail the run.** A malformed expression is rejected when you register the workflow. At run time, any evaluation error — a missing key, a non-string result, or an empty result — silently falls back to the generated `<readableId|workflowName>-<timestamp>` name. A result longer than 255 characters is stored truncated (rune-safe), never rejected.

</Callout>

## Declaring an expression

Set the expression on the workflow and/or task definition.

<UniversalTabs items={['Python', 'Typescript', 'Go', "Ruby"]}>
  <Tabs.Tab>

```python
# Workflow-level: names the run from the run input
customer_workflow = hatchet.workflow(
    name="CustomerWorkflow",
    display_name="input.customerName",
    input_validator=WorkflowInput,
)

# Per-task: names an individual DAG step
@customer_workflow.task(display_name="'enrich-' + input.customerName")
def enrich(input: WorkflowInput, ctx: Context) -> None:
    ...
```

// Workflow-level: names the run from the run input
export const customerWorkflow = hatchet.workflow<WorkflowInput, WorkflowOutput>({

## Fan-out and bulk runs

Because the name is derived from each run's own input, a `run_many` / bulk trigger — or a parent fanning out many children — produces a distinct, meaningful name per run without any per-item configuration, directly solving the identical-rows problem described above.

## Event- and cron-triggered runs

The same expression applies to runs started by an [event](./events.mdx) or a [cron schedule](./scheduled-runs.mdx): the event's or cron's input payload is the `input` the expression reads, so those runs get evaluated names too.
