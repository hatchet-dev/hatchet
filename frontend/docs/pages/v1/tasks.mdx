# Tasks

The fundamental unit of work in Hatchet is a **task**. At its most basic level, a task is just a function. You can invoke a task on its own (a "standalone" task), or compose tasks into a larger [workflow](/v1/workflows), which we'll discuss in more detail later on in the documentation.

## Defining a task

All that's needed to define a task in Hatchet is a function and a name (only required in some SDKs). At the most basic level, a task is a function that accepts some [input](#input-and-output), and a [context](#the-context-object).

## Input and output

Every task receives an **input** - a JSON-serializable object passed when the task is triggered. The value that is returned from the task becomes the task's **output**, which callers receive when they await the result of the task.

Hatchet's SDKs support type-checked and runtime-validated input and output types for tasks, so that you can integrate your Hatchet tasks into your codebase in a type-safe and predictable way that provides you all of the guarantees you get from, for example, replacing the Hatchet task run with a local function call.

You can refer to the [examples above](#defining-a-task) to see how to provide validators for task inputs and outputs.

## The context object

In addition to input and output payloads, every task receives a **context**. The context provides Hatchet-related information that might be useful to the execution of the task at runtime. For instance, you might access the `workflow_run_id`, the `task_run_id`, or the `retry_count` from the context and have your task's application logic do something with those values.

The context also provides a helper methods for interacting with a number of Hatchet's features, such as [managing cancellations](/v1/cancellation), [refreshing timeouts](/v1/timeouts#refreshing-timeouts), [pushing stream events](/v1/streaming#pushing-stream-events) and more.

## Configuration

You can configure tasks to handle common problems in queueing systems. Here are a few common examples, with a solution via task configuration:

### I want my task to retry if it fails

You can retry tasks automatically on failure by configuring a [retry policy](/v1/retry-policies) on your task. This will tell Hatchet to retry your task up to the number of times you've allowed until it succeeds.

### I want to limit my customers to one running task at a time

You can utilize Hatchet's [concurrency](/v1/concurrency) features to define the behavior you'd like, depending on the needs of your application. For instance, if you have a customer performing some action and you'd like to process those actions one at a time, you might use a `GROUP_ROUND_ROBIN` strategy. If you'd like any in-progress work to be cancelled when a new task for that customer comes in, you can use `CANCEL_IN_PROGRESS`. And similarly, if you'd like to drop any new work on the floor if a task is already running for that customer, you can use `CANCEL_NEWEST`.

### I want my tasks to fail if they take too long

Hatchet supports a number of [timeout](/v1/timeouts) features, which allow your tasks to fail after a specified amount of time either in the queue or running.

### I have a machine learning model loaded on a specific worker, and I need a task to run on _that_ worker

You can route tasks to workers using [worker affinity](/v1/advanced-assignment/worker-affinity), which assigns tasks to workers based on matching labels. This can be done declaratively by providing the labels when the task is created, or dynamically at runtime.

## How tasks execute on workers

Tasks don't run on their own. [Workers](/v1/workers) execute them. A worker is a long-running process that registers one or more tasks with Hatchet. When you trigger a task, Hatchet places it in a queue and assigns it to an available worker that has registered that task. When a task completes, the Hatchet SDK running on the worker sends the result of the task back to the Hatchet Engine, so the engine can mark the task as a success or failure, display the results, and so on.
