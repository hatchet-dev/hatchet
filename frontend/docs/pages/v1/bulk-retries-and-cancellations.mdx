# Replays

Replays let you re-run failed or canceled task runs. You can replay individual runs from the Dashboard, or replay many runs at once via the SDKs, REST API, or Dashboard.

## Replaying from the Dashboard

{/* TODO: Add a video of the dashboard view */}

Select runs from the runs list and click "Replay" in the toolbar, or open a specific run to inspect it and replay from there. This is useful when you want to wait for an external dependency to recover, or when you need to deploy a fix before retrying affected runs.

## Programmatic Bulk Replays

You can replay many runs at once via the SDKs and REST API. There are two approaches:

1. [**By run IDs**](#by-run-ids) - provide a list of task run IDs to replay.
2. [**By filters**](#by-filters) - match runs using the same filters available in the Dashboard. For example, replay all failed runs of `SimpleTask` from the past fifteen minutes where `additional_metadata.foo` equals `bar`.

## A Note on Dead Letter Queues

In messaging systems like Kafka or SQS, a dead letter queue (DLQ) captures messages that fail processing so they can be handled manually.

Hatchet does not have a separate DLQ. Instead, failed runs are persisted in the Dashboard where you can inspect them, identify root causes, and replay them once the underlying issue is resolved. We often see developers creating scripts and tooling around the bulk operations API to automate these recovery workflows.
