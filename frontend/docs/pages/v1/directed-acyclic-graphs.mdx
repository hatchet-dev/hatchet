# DAGs as Durable Workflows

A directed acyclic graph (DAG) is a workflow where every task and every dependency between tasks is declared upfront, before the workflow runs. You describe the shape of the work — which tasks exist, which ones depend on which — and Hatchet handles the rest: execution order, parallelism for independent tasks, passing outputs from parents to children, dashboard visibility, and so on.

<WorkflowDiagram />

DAGs in Hatchet are, by definition, a form of [durable execution](/v1/durable-execution). Each task in a DAG is effectively a checkpoint: once a task completes, Hatchet writes its output to the durable event log, and if the workflow is interrupted or a later task has to retry, Hatchet does not re-run any task that has already finished successfully. You get the same exactly-once-style guarantees as a [durable task](/v1/durable-tasks), just with the shape of the work declared ahead of time rather than decided at runtime.

What makes DAGs particularly nice is that you get those durable properties _alongside_ a set of workflow-building features that you'd otherwise have to construct by hand: automatic parallelism between independent tasks, typed inputs and outputs flowing between parents and children, [parent-driven branching](#branching-with-parent-conditions), [or groups](#waiting-on-conditions-with-or-groups) for combining sleeps, events, and parent outputs into arbitrary wait conditions, and a clear, visual representation of the workflow in the Hatchet dashboard. The result is that for any workload whose structure is known upfront, a DAG is usually the most ergonomic way to get durable execution in Hatchet.

Use a DAG when you know the shape of the work before it starts. ETL pipelines, document processing, CI/CD-style jobs, and other fixed multi-stage workflows are all good fits. If the shape of the work _isn't_ known upfront — for example, if you need to spawn children based on runtime data, or loop until some condition is met — reach for [durable tasks](/v1/durable-tasks) instead.

## Adding task dependencies

The real power of a DAG comes from connecting tasks. A task can declare one or more **parent** tasks — those parents must complete successfully before this task starts, and any tasks that don't have a dependency relationship between them are run in parallel. When a task runs, its parents' outputs are available on the context object, so data flows naturally from one stage of the DAG to the next.

## Running a workflow

Once you've defined a workflow and registered it on a [worker](/v1/workers), you can trigger it the same way you'd trigger a standalone task — run and wait for a result, enqueue it and let it run in the background, schedule it for the future, and so on. See the [Running Tasks](/v1/running-your-task) documentation for the full set of options.

## Branching with parent conditions

Even though a DAG's structure is fixed at definition time, individual tasks can decide whether to run based on data produced earlier in the workflow. **Parent conditions** let a task inspect a parent's output and either skip or cancel itself based on what it finds. There are two operators available:

- **`skip_if`** — skip this task if the parent's output matches the condition. Downstream tasks can [check whether a parent was skipped](#checking-if-a-parent-was-skipped) and react accordingly.
- **`cancel_if`** — cancel this task if the parent's output matches the condition.

<Callout type="warning">
  A task cancelled by `cancel_if` behaves like any other cancellation in
  Hatchet, meaning its downstream dependents will be cancelled as well.
</Callout>

The most common use for parent conditions is to express sibling branches in a DAG. You declare a base task that returns some data, then define two sibling tasks with complementary `skip_if` conditions so that only one of the two runs per workflow execution.

First, declare the base task that returns a value:

## Waiting on conditions with or groups

In addition to parent conditions, a task in a DAG can wait for external signals before it runs: a sleep timer to expire, a [user event](/v1/events) to arrive, or some combination of these alongside parent conditions. You compose these together using **or groups**.

An or group is a set of conditions combined with an `Or` operator, which evaluates to `True` if **at least one** of its conditions is satisfied. Multiple or groups on the same task are combined with `AND`, so every group must have at least one satisfied condition for the task to proceed. Taken together, this lets you express arbitrarily complex sets of conditions in [conjunctive normal form](https://en.wikipedia.org/wiki/Conjunctive_normal_form) (CNF).

### Sleep + event example

The most common combination is a sleep condition alongside an event condition: proceed when an external signal arrives _or_ after a timeout, whichever comes first. This is a natural fit for human-in-the-loop workflows where you want a deadline on how long you'll wait for a response.

### Combining multiple or groups

For more complex logic, you can declare multiple or groups on a single task. As an example, consider these three conditions:

- **Condition A**: A parent task's output is greater than 50
- **Condition B**: A 30 second sleep timer expires
- **Condition C**: A `payment:processed` event arrives

To proceed when `(A or B)` **and** `(A or C)` are both satisfied, you'd declare two separate or groups: one containing `A or B`, and the other containing `A or C`. The task runs once both groups are satisfied. If `A` is true, both groups pass immediately. If `A` is false, the task needs both `B` (the sleep expires) and `C` (the event arrives) before it can run.

### Common combinations

A few combinations of conditions come up often enough to be worth highlighting:

| Combination    | Use case                                                                             |
| -------------- | ------------------------------------------------------------------------------------ |
| Sleep + Event  | Proceed after a timeout _or_ when an external signal arrives (whichever comes first) |
| Parent + Event | Proceed if a parent output meets a threshold _or_ a manual override event arrives    |
| Parent + Sleep | Proceed if a parent indicates readiness _or_ after a maximum wait time               |
| All three      | Complex gates combining data-driven, time-based, and event-driven conditions         |

## Fixed pipelines

DAGs naturally model multi-stage pipelines where every step is known ahead of time. ETL workflows, document processing pipelines, and CI/CD-style workflows are all good examples — each stage depends on the previous one, and the overall structure is visible and predictable in the dashboard.
