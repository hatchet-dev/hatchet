# Webhooks

<Callout type="info" emoji="🪓">
  This feature is currently in development and may change. Reach out with
  feedback or if any problems are encountered registering external webhooks.
</Callout>

Webhooks allow external systems to trigger Hatchet workflows by sending HTTP requests to dedicated endpoints, enabling real-time integration with third-party services or any system capable of sending webhook events.

## Creating a Webhook

Creating a webhook requires configuring a set of fields that tell Hatchet which workflows to trigger and how to validate the incoming request. The required fields are described below.

#### Name

The **Webhook Name** is tenant-unique and is used to construct the endpoint URL for incoming requests. For example, a tenant with ID `d60181b7-da6c-4d4c-92ec-8aa0fc74b3e5` and a webhook named `my-webhook` would produce a URL like `https://cloud.onhatchet.run/api/v1/stable/tenants/d60181b7-da6c-4d4c-92ec-8aa0fc74b3e5/webhooks/my-webhook`. The URL can be copied directly from the dashboard.

#### Source

The **Source** identifies the origin of the webhook. Pre-configured sources (GitHub, Stripe, Slack, Linear, and Svix) provide sensible defaults for easy setup. A "generic" source allows all fields to be configured manually based on what the webhook sender provides.

#### Event Key Expression

The **Event Key Expression** is a [CEL](https://cel.dev/) expression that produces a dynamic event key from the incoming webhook payload and headers. It can be a constant (e.g. `'webhook'`) or a dynamic expression. Examples:

1. `'stripe:' + input.type`: prefixes all keys with `'stripe:'` and appends the payload's `type` field, producing keys like `stripe:payment_intent.created`.
2. `'github:' + headers['x-github-event'] + ':' + input.action`: produces keys like `github:star:created`.

<Callout type="info">
  The event key expression result is what Hatchet uses to match workflows. A
  workflow must declare a matching event key as a trigger (for example,
  `on_events=["stripe:payment_intent.created"]`) to be triggered by the incoming
  webhook.
</Callout>

#### Scope Expression (Optional)

The **Scope Expression** is an optional [CEL](https://cel.dev/) expression that evaluates to a string used to filter which workflows to trigger. This is useful when multiple workflows listen to the same event key and routing based on payload content is required.

Both `input` (the webhook payload) and `headers` (the request headers) are available. Examples:

1. `input.customer_id`: uses the customer ID from the payload as the scope.
2. `headers['x-organization-id']`: uses a header value as the scope.
3. `input.metadata.environment`: routes to different workflows based on environment.

#### Static Payload (Optional)

The **Static Payload** is an optional JSON object merged with the incoming webhook payload before it is passed to workflows. Use cases include:

- Adding constant metadata to all events from this webhook.
- Injecting configuration values absent from the original payload.
- Overriding specific fields from the incoming payload.

<Callout type="info">
  When a key collision occurs between the incoming payload and the static
  payload, the static payload value takes precedence.
</Callout>

For example, with a static payload of `{"source": "stripe", "environment": "production"}` and an incoming payload of `{"type": "payment_intent.created", "source": "api"}`, the final payload delivered to the workflow is `{"type": "payment_intent.created", "source": "stripe", "environment": "production"}`.

#### Authentication

Authentication specifies how Hatchet validates incoming webhook requests. For [pre-configured sources](#pre-configured-sources), Hatchet provides presets for most fields. Typically only a secret is required.

For generic sources, an authentication method must be selected (basic auth, API key, or HMAC) along with its required fields.

<Callout type="warning">
  Hatchet encrypts all secrets provided for validating incoming webhooks.
</Callout>

Required fields per authentication method:

- **Pre-configured sources**: webhook secret only.
- **Generic sources**:
  - **Basic Auth**: username and password.
  - **API Key**: header name containing the key, and the secret key.
  - **HMAC**: header name containing the signature, the secret, an encoding method (e.g. `hex`, `base64`), and an algorithm (e.g. `SHA256`, `SHA1`).

## Usage

The webhook URL is available during and after creation. This is the URL provided to the webhook sender.

The final step is registering the event keys that workers should listen for. See the [documentation on event triggers](/v1/external-events/run-on-event) for examples.
