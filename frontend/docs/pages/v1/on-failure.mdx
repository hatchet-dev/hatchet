# Error Handling

When a task fails, you often want to run cleanup logic, send a notification, or trigger a compensating action. Durable tasks handle this with your language's normal error handling; DAGs provide a dedicated on-failure task.

## Handling errors in durable tasks

Durable tasks are just functions, so you handle errors with the same tools you use everywhere else — `try`/`except` in Python, `try`/`catch` in TypeScript and Go. You have full control over what happens when a child task or operation fails.

Some things you can do in a catch block:

- **Retry with backoff** — sleep for a bit and retry the child task.
- **Fall back to an alternative** — if a primary path fails, spawn a different child task.
- **Handle partial failures** — in a fan-out, collect results from the successful children and handle failures individually, rather than failing the whole workflow.
- **Clean up** — release resources, cancel in-progress work, or notify external systems.

## On-failure tasks

The on-failure task is a special task that runs when any task in the workflow fails. It lets you handle errors, perform cleanup, or send notifications as a first-class part of the workflow definition.

### Defining an on-failure task

The on-failure task will run if any of the main tasks in the workflow fail. A few things you might use it for:

- Cleaning up resources or partial state after a failure
- Sending a notification or alert
- Logging extra context for debugging
- Kicking off a compensating action or fallback workflow
