# Directed Acyclic Graphs (DAGs)

If you know every step your workflow will take before it starts, you can define your workflow as a directed acyclic graph (DAG). DAGs let you declare tasks and dependencies upfront, and then Hatchet handles the rest: Execution order, parallelism for independent tasks, passing outputs downstream, and dashboard visibility.

## Defining a task

Now that we have a workflow, we can add tasks to it. Each task is a function that receives the workflow's input and optionally returns some output. Tasks also accept a number of configuration options: [Retry policies, timeouts, concurrency settings, and so on](/v1/tasks#configuration). For more details on defining tasks, see the [tasks documentation](/v1/tasks).

## Adding task dependencies

The real power of a DAG workflow comes from connecting tasks. A task can declare which other tasks it depends on — those tasks must complete before this one starts. Outputs from parent tasks are automatically available in child tasks.

## Fixed pipelines

DAGs naturally model multi-stage pipelines where every step is known ahead of time. ETL workflows, document processing pipelines, and CI/CD workflows are all good examples — each stage depends on the previous one, and the overall structure is visible and predictable in the dashboard.
