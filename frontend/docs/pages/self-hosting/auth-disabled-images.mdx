# Auth-disabled dev images

Hatchet publishes a set of dev images (`hatchet-lite-dev`, `hatchet-api-dev`, `hatchet-engine-dev`, `hatchet-admin-dev`) built with authentication compiled out, so you can run a local instance without signing in to the dashboard or creating your own API token.

<Callout type="error" emoji="🚫">
  The dev images are for local development and testing only.
</Callout>

## What the dev images do

- **Dashboard** loads without a login screen; requests run as the seeded admin user (`ADMIN_EMAIL`, default `admin@example.com`).
- **REST API** calls succeed without a bearer token or session cookie.
- **Single tenant.** Everything runs against the seed default tenant, so tenant creation, signup and invites are disabled (and hidden in the dashboard).

## Workers still use an API token

The gRPC engine is unchanged - workers and SDK clients still authenticate with an API token. The dev images ship a single, fixed token that is the same on every auth-disabled instance:

- The token (and the keyset that signs it) is embedded (via `go:embed`) only in dev builds. The regular images never compile it in, so the token is inert against any normal deployment.
- It is scoped to the seed default tenant and does not expire.
- View it in the dashboard (the red banner has a **View worker token** button, and it is prefilled in the onboarding "Set your profile" step), or fetch it from the running container.

Point your worker at it via `HATCHET_CLIENT_TOKEN`.

## Running it

The easiest way is the Hatchet CLI's `--disable-auth` flag, which starts a Postgres container and the `hatchet-lite-dev` image for you:

```sh
hatchet server start --disable-auth
```

To run it yourself, follow the [Hatchet Lite](/self-hosting/hatchet-lite) guide but swap the image for `ghcr.io/hatchet-dev/hatchet/hatchet-lite-dev:latest`. The dev image still needs an external Postgres and the usual server configuration - the only difference is that authentication is compiled out.

On startup the container prints the embedded worker token and writes it to the config volume. Grab it from the logs or the volume:

```sh
docker logs <hatchet-lite-container> # printed under "authdisabled build: worker API token"
# or
docker exec <hatchet-lite-container> cat ./config/authdisabled-token
```

Then point your worker at it:

```sh
export HATCHET_CLIENT_TOKEN="<token>"
```

The `hatchet-api`, `hatchet-engine` and `hatchet-admin` images have dev variants too (`ghcr.io/hatchet-dev/hatchet/hatchet-<component>-dev`).
